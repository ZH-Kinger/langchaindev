# 飞书 Bot 智能运维助手 — 开发 Story

**项目代号**：AIOps-FeishuBot | **版本**：v1.0 | **负责人**：王梓涵 | **更新日期**：2026-04-27

---

## 一、背景与目的

GPU 算力资源申请流程分散，用户需要在多个系统（Jira、PAI 控制台、飞书）之间切换，审批链路不透明，资源空转问题无人感知。

本项目目标：

- 将 GPU 申请、审批、监控、停止全部收敛到飞书群内完成
- 通过 Prometheus 指标自动识别 GPU 空转，规范化 GPU 利用率管理，减少算力浪费
- 降低 SRE 人工介入成本，申请到实例就绪全程自动化

---

## 二、需求分析

### 核心场景

| 场景 | 当前问题 | 期望行为 |
|------|---------|---------|
| GPU 申请 | 手动填 Jira + 等人审批，流程不透明 | 飞书卡片提交，自动创建工单，审批结果实时通知 |
| 实例管理 | 需登录 PAI 控制台查看 | 飞书内查询状态、延续、停止 |
| 空转检测 | 无主动感知，资源浪费无人处理 | SM 利用率低于阈值持续超限，自动推送提醒 |
| 超时管理 | 实例到期无提醒，费用持续产生 | 到期前 15 分钟推送续期/停止卡片，无响应自动停止 |
| 大规格审批 | 邮件审批，响应慢 | 飞书卡片一键批准/拒绝 |

### 目前存在的问题

1. GPU 空转阈值硬编码，不区分 GPU 型号（A10/A100/H100 的 TDP 和温度上限不同）
2. 趋势告警缺失：利用率持续下降时无主动通知，训练卡住只能靠用户自查
3. 多实例场景下缺少横向对比，用户无法快速判断哪个实例值得优先处理
4. `_advice_thermal` 依赖 regex 解析格式化字符串提取温度，措辞变更会静默失效
5. 集群级建议和实例级分析的 Agent 路由边界模糊，关键词存在重叠

### GPU 利用率规范化验收

| 指标 | 目标值 |
|------|--------|
| 空转实例（SM 利用率 < 10% 持续 30 分钟）检出率 | ≥ 95% |
| 趋势告警从触发到飞书送达延迟 | < 2 分钟 |
| 热管理误报率（阈值按型号区分后） | < 5% |
| 多实例横向对比报告覆盖率 | 100%（≥ 2 实例时必须输出） |

## 六、非功能性要求

| 类别 | 要求 |
|------|------|
| 响应时延 | 飞书消息响应 < 5s，Agent 复杂调用 < 30s |
| 安全 | Verification Token 校验防伪造；AK/SK 建议加密存储 |
| 可用性 | Redis 故障时 Bot 基础回复仍可用，配额/状态降级处理 |
| 部署 | 公网 HTTPS 可达（nginx 反代 + SSL） |

---

## 七、风险与依赖

| 风险项 | 影响 | 缓解措施 |
|--------|------|---------|
| 飞书回调地址不可达 | Bot 完全不可用 | ngrok（开发）/ nginx+HTTPS（生产） |
| Redis 宕机 | 配额/GPU 状态丢失 | AOF 持久化 + 优雅降级 |
| PAI DSW API 变更 | 实例创建失败 | SDK 版本锁定 + 异常重试 |
| Jira 网络中断 | 工单创建失败 | 本地队列暂存 + 重试 |
| Prometheus 指标缺失 | 空转/趋势检测失效 | 降级为静态阈值告警 |
